# Backend API — Endpoint Reference

All authenticated endpoints use **Bearer token** authentication.  
Pass the API key in the `Authorization` header:

```
Authorization: Bearer <api_key>
```

API keys are **256-bit random hex strings** generated by the server (shown once at user creation).

---

## `GET /api/health`

**Auth:** ❌ None  
**Response:** `200 OK` — plain text `ok`

Simple liveness check.

---

## `GET /api/auth/me`

**Auth:** ✅ Sudo privileges required  
**Response:** `200 OK`

Returns the currently authenticated user's profile.

```json
{
  "user": {
    "id": "97f6dd1e-39ad-4101-9323-69f26e6e969a",
    "email": "admin@example.com",
    "is_sudoer": true,
    "created_at": "2026-05-14 07:32:23"
  }
}
```

---

## `POST /api/auth/register`

**Auth:** ✅ Sudo privileges required  
**Content-Type:** `application/json`

Creates a new **regular** (non-sudo) user. The API key is returned **once** — it will not be shown again.

### Request body

```json
{
  "email": "user@example.com"
}
```

### Response `201 Created`

```json
{
  "api_key": "f7a588bf254892789e91353b4a37646b8dfeba601ac6651746ea9dc43c582374",
  "user": {
    "id": "21f85918-1b43-4a66-b446-023ae159f8d4",
    "email": "user@example.com",
    "is_sudoer": false,
    "created_at": "2026-05-14 07:41:46"
  }
}
```

### Errors

| Status | Meaning |
|--------|---------|
| `403` | Caller is not a sudoer |
| `409` | Email already registered |

---

## `GET /api/settings`

**Auth:** ✅ Sudo privileges required  
**Response:** `200 OK`

Returns the current backend settings.

```json
{
  "fail2ban_max_attempts": 5
}
```

---

## `GET /api/settings/schema`

**Auth:** ✅ Sudo privileges required  
**Response:** `200 OK`

Returns a [JSON Schema (draft 2020-12)](https://json-schema.org/) describing the settings object.  
Frontends can use this to dynamically render a settings form with proper field types, descriptions, and validation constraints.

```json
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "type": "object",
  "title": "AppSettings",
  "properties": {
    "fail2ban_max_attempts": {
      "type": "integer",
      "format": "uint32",
      "minimum": 0,
      "description": "Max failed auth attempts before an IP is blacklisted permanently."
    }
  },
  "required": ["fail2ban_max_attempts"]
}
```

---

## `PUT /api/settings`

**Auth:** ✅ Sudo privileges required  
**Content-Type:** `application/json`

Update backend settings. The changes are persisted to `data/settings.json` and take effect immediately.

### Request body

```json
{
  "fail2ban_max_attempts": 10
}
```

### Response `200 OK`

```json
{
  "fail2ban_max_attempts": 10
}
```

### Errors

| Status | Meaning |
|--------|---------|
| `403` | Caller is not a sudoer |

---

## `GET /api/ipban/status`

**Auth:** ✅ Sudo privileges required  
**Response:** `200 OK`

Returns the current state of the IP blacklist and fail2ban configuration.

```json
{
  "blacklisted_ips": [
    "192.168.1.100",
    "10.0.0.99"
  ],
  "max_attempts": 5
}
```

---

## `POST /api/ipban/blacklist`

**Auth:** ✅ Sudo privileges required  
**Content-Type:** `application/json`

Add an IP address to the permanent blacklist.  
The IP is immediately banned and added to `data/blacklist.json` for persistence across restarts.

### Request body

```json
{
  "ip": "192.168.1.100"
}
```

### Response `200 OK`

```json
{
  "blacklisted": true,
  "ip": "192.168.1.100"
}
```

### Errors

| Status | Meaning |
|--------|---------|
| `403` | Caller is not a sudoer |

---

## `POST /api/ipban/unban`

**Auth:** ✅ Sudo privileges required  
**Content-Type:** `application/json`

Remove an IP from the blacklist (whether it was added manually or by fail2ban).

### Request body

```json
{
  "ip": "192.168.1.100"
}
```

### Response `200 OK`

```json
{
  "unbanned": true,
  "ip": "192.168.1.100"
}
```

`"unbanned": false` means the IP was not in the blacklist.

### Errors

| Status | Meaning |
|--------|---------|
| `403` | Caller is not a sudoer |

## `POST /api/auth/register` (CLI equivalent: `backend create-sudo`)

Only available as a CLI command on the host machine. Creates a sudo (admin) user directly in the database.

```
backend create-sudo --email admin@example.com
```

Output:

```
✅ Sudo user created successfully!
   Email:   admin@example.com
   API key: 8524d83ac0ae0e9da4938ff46dc04c10a754ca58be529ff2c5d2602e8aba4327

⚠️  Save this API key — it will not be shown again.
```

## CLI Commands Reference

```
Minecraft server backend

Usage: backend <COMMAND>

Commands:
  serve        Start the HTTP API server
  create-sudo  Create a new sudo (admin) user directly in the database
  list-users   List all users in the database
  ban-status   View fail2ban / blacklist status
  unban        Remove an IP from the blacklist
  help         Print this message or the help of the given subcommand(s)
```

### `backend serve`

Starts the HTTP API server listening on `0.0.0.0:3000`.  
Reads `data/settings.json` and `data/blacklist.json` on startup.

```
RUST_LOG=info backend serve
```

### `backend list-users`

```
ID         Email                          Sudo   Created
---------------------------------------------------------------------------
582e2c66   admin@example.com              yes    2026-05-14 07:41:42
21f85918   user@example.com               no     2026-05-14 07:41:46
```

### `backend ban-status`

Displays the current fail2ban configuration and all blacklisted IPs.

### `backend unban <ip>`

Removes an IP from `data/blacklist.json`.

```
backend unban 192.168.1.100
```

## File Layout

```
data/
├── app.db              # SQLite database (users, etc.)
├── settings.json       # Backend settings (fail2ban_max_attempts)
└── blacklist.json      # JSON array of blacklisted IPs
```

---

## Authentication Flow

1. **Sudo user creation** — only via `backend create-sudo` on the host machine
2. **Regular user creation** — sudoer calls `POST /api/auth/register`, gets the new user's API key
3. **All API calls** — use `Authorization: Bearer <api_key>` header

No login/logout endpoints — the API key **is** the credential.

---

## Fail2ban

The blacklist is stored in `data/blacklist.json` as a plain JSON array:

```json
["192.168.1.100", "10.0.0.99"]
```

Fail2ban monitors failed authentication attempts per IP using a 10-minute sliding window.  
When `fail2ban_max_attempts` is exceeded, the IP is added to the blacklist immediately and permanently.
Only a sudoer can remove it (via API or CLI).

## `mc-server-manager` Crate

A separate library crate at `mc-server-manager/` for spawning, monitoring, and
controlling Minecraft Java Edition server processes.

### Usage

```rust
use mc_server_manager::{ServerConfig, ServerInstance};

let config = ServerConfig::new(
    "/srv/minecraft/paper-1.21.jar",   // server JAR
    "/usr/bin/java",                     // Java binary
    "512M",                              // min heap
    "4G",                                // max heap
    "/srv/minecraft",                    // working directory
)
.with_jvm_args(vec!["-XX:+UseG1GC".into()]);

let mut server = ServerInstance::start(config).await.unwrap();

// Read stdout line by line
while let Some(line) = server.stdout_line().await {
    println!("[MC] {line}");
}

// Send commands server.send_command("say Hello everyone!").unwrap();

// Graceful stop
server.stop().await.unwrap();
```

### API

| Method | Description |
|--------|-------------|
| `ServerConfig::new(...)` | Configure a server instance |
| `ServerInstance::start(cfg)` | Launch the JVM process |
| `server.send_command(cmd)` | Write to server stdin |
| `server.stop()` | Send `stop` and wait for exit |
| `server.kill()` | Force-kill the process |
| `server.is_running()` | Check if process is alive |
| `server.stdout_line()` | Read next line of server stdout |

### Git submodule

The crate is designed to be published independently (or used as a git
submodule). Add it to your own project with:

```toml
[dependencies]
mc-server-manager = { path = "../mc-server-manager" }
```

Or from the monorepo root, it is already a workspace member — compile
with `cargo build` from the root.
